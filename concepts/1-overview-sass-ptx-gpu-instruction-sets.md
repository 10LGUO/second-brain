# SASS and PTX — GPU Instruction Sets

NVIDIA GPUs use a two-level instruction set architecture: **PTX** (a virtual, portable instruction set) and **SASS** (the actual native machine code). Understanding these levels is important for deep compiler and operator work, especially for anyone working at the bottom layers of the AI infra stack.

## Key Properties

### PTX (Parallel Thread Execution)

- **Virtual ISA**: PTX is a stable, portable intermediate representation that NVIDIA exposes publicly. It is analogous to LLVM IR or Java bytecode — it describes computation without being tied to a specific hardware generation.
- **Stable across generations**: Code written or compiled to PTX can run on future GPU architectures. The driver JIT-compiles PTX to native SASS at load time.
- **Human-readable**: PTX is text-based and relatively legible. Registers are typed (`.f32`, `.f64`, `.u32`, `.pred`, etc.) and instructions map closely to GPU operations.
- **Source of truth for CUDA**: When you compile a CUDA `.cu` file with `nvcc`, it first lowers to PTX, then optionally embeds both PTX and a pre-compiled SASS binary (called a "cubin") in the final `.fatbin` executable.
- **Forward compatibility**: If only PTX is embedded (no cubin for a target architecture), the driver compiles it at runtime — this is called "JIT compilation from PTX."
- **Typed register file**: PTX enforces register types at the IR level, which allows the backend compiler (`ptxas`) to make register allocation decisions per architecture without burdening the frontend compiler.
- **Predicated execution**: PTX exposes predicate registers (`.pred`) that allow conditional execution of instructions without branches, mapping directly to hardware-level predication in SASS.

### SASS (Shader ASSembly)

- **Native machine code**: SASS is the actual binary instruction set executed by the GPU hardware. It is architecture-specific — Volta SASS differs from Ampere SASS, which differs from Hopper SASS.
- **Not publicly documented**: Unlike PTX, NVIDIA does not fully document SASS. Knowledge of it comes from reverse engineering (e.g., the `nvdisasm` tool, the Turingas/Maxas assembler projects, and community research).
- **Architecture-versioned**: SASS binaries are tagged with a compute capability (e.g., `sm_80` for Ampere, `sm_90` for Hopper). A binary compiled for `sm_80` will not run on `sm_70`.
- **Inspection with nvdisasm**: NVIDIA ships `nvdisasm` as part of the CUDA toolkit, which can disassemble cubin files into human-readable SASS. This is the primary official window into what the hardware actually executes.
- **Performance-critical details**: SASS exposes scheduling slots, predication, register reuse flags, and memory access patterns that are invisible at the PTX level. Achieving peak performance often requires inspecting SASS output.
- **Instruction scheduling**: SASS includes explicit control bits that encode instruction-level scheduling decisions — e.g., stall counts, yield flags, read-after-write barriers, and warp yield hints. These are generated by `ptxas` and are invisible at the PTX level.
- **Register reuse flags**: SASS encodes register reuse hints that tell the hardware to keep a register value in the register file operand collector rather than reading from the main register file again, reducing energy and latency. These optimizations are entirely absent from PTX.

### The Compilation Pipeline

```text
CUDA C/C++ (.cu)
      │
      ▼  nvcc (frontend / driver compiler)
    PTX  ←── also exposed here (--ptx flag)
      │
      ▼  ptxas (PTX assembler / backend compiler)
    SASS / cubin  ←── embedded in fatbin
      │
      ▼  GPU driver (optional JIT if cubin missing)
      │
      ▼  GPU hardware executes SASS
```

- `nvcc --ptx` dumps PTX for inspection.
- `nvcc --cubin` compiles all the way to a SASS binary.
- `cuobjdump --dump-sass <binary>` or `nvdisasm <cubin>` reveals the SASS.
- The driver can also JIT-compile PTX → SASS at program startup if a cubin for the current architecture is not found in the fatbin.
- `ptxas` is the component that performs register allocation, instruction scheduling, and control-flow lowering. Its optimization quality varies by architecture and CUDA version, and it can be invoked standalone.
- The fatbin format allows a single executable to embed multiple cubins (e.g., one for `sm_80` and one for `sm_90`) plus a PTX fallback, enabling broad deployment across GPU generations.

## Variants and Related Concepts

### Compute Capability Targets

Each NVIDIA GPU generation maps to a compute capability version:

| Architecture | Compute Capability | Common GPUs |
| --- | --- | --- |
| Volta | sm_70 | V100 |
| Turing | sm_75 | T4, RTX 2080 |
| Ampere | sm_80 / sm_86 | A100, A10, RTX 3090 |
| Ada Lovelace | sm_89 | RTX 4090, L4 |
| Hopper | sm_90 / sm_90a | H100 |
| Blackwell | sm_100 / sm_100a | B100, B200 |

PTX targets a virtual compute capability (e.g., `compute_90`); SASS targets a real hardware SM version (e.g., `sm_90`). The `a` suffix (e.g., `sm_90a`) indicates architecture-specific extensions such as Hopper's wgmma and TMA instructions, which are not available through the generic `sm_90` target.

### Inline PTX in CUDA

CUDA C++ allows embedding raw PTX directly via `asm volatile(...)` blocks, similar to inline assembly in C. This is used in performance-critical kernels (e.g., in FlashAttention, cuBLAS) to guarantee specific instruction sequences or to access hardware features not exposed through CUDA intrinsics.

```cuda
// Example: load with cache-evict hint (streaming load, bypass L1)
asm volatile("ld.global.cs.f32 %0, [%1];" : "=f"(val) : "l"(ptr));

// Example: warp shuffle (often faster than intrinsic wrappers)
asm volatile("shfl.sync.bfly.b32 %0, %1, 0x10, 0x1f, %2;"
             : "=r"(out) : "r"(in), "r"(mask));

// Example: Hopper async bulk copy via TMA (requires sm_90a target)
asm volatile("cp.async.bulk.tensor.1d.shared::cluster.global.mbarrier::complete_tx::bytes"
             " [%0], [%1, {%2}], %3, [%4];"
             :: "r"(dst), "l"(tma_desc), "r"(coord), "r"(size), "r"(mbar));
```

Inline PTX usage is common in libraries like FlashAttention-2/3, TransformerEngine, and cuBLAS for precise control over memory access semantics and instruction selection. The compiler cannot reorder or optimize across `asm volatile` boundaries, which is both its strength (predictability) and its weakness (prevents inter-statement optimization).

### Triton and PTX/SASS

[[triton]] compiles Python kernel definitions → Triton IR → LLVM IR → PTX → SASS. Inspecting PTX and SASS output from Triton kernels is a standard debugging technique for understanding why a kernel is slower than expected. The `triton.tools.disassemble` utilities and the `TRITON_PRINT_AUTOTUNING` env flags assist with this.

Key Triton-specific notes:

- Triton's LLVM backend targets PTX, not SASS directly; `ptxas` handles the final lowering.
- Triton kernels can be inspected via `kernel.asm['ptx']` and `kernel.asm['sass']` after compilation, enabling direct comparison.
- Common Triton SASS issues include register spilling (visible as `ld`/`st` to local memory in SASS), excessive predication overhead, and suboptimal memory instruction selection.
- Triton 3.x introduced hooks for injecting inline PTX into Triton-generated kernels, bridging the gap between Triton's high-level abstractions and hardware-specific instruction sequences.

### Custom SASS Assemblers

Because NVIDIA does not publish a SASS assembler, the community has built tools like:

- **Maxas** (Maxwell assembler): early pioneer in hand-written SASS for sm_50/sm_52, demonstrated near-peak SGEMM performance.
- **Turingas**: extended the Maxas approach to Turing (sm_75).
- **CuAssembler**: more recent community assembler covering Turing and Ampere (sm_75 through sm_86), with partial Hopper support in development.
- **as-hack in cuBLAS**: NVIDIA's own libraries reportedly use internal SASS-level tricks not reproducible from PTX, likely through internal assembler tooling not distributed with the CUDA toolkit.
- **CUTLASS and CuTe**: While not SASS assemblers, CUTLASS and the CuTe layout algebra library are NVIDIA's preferred abstraction for hand-tuned kernels that produce near-optimal SASS; they work at the PTX intrinsic level using CUDA's `mma` and `cp.async` intrinsics.

These tools are used by researchers trying to achieve theoretical peak FLOPS, e.g., for matrix multiply or memory copy routines.

### PTX vs SASS: What Gets Lost in Translation

Understanding what `ptxas` adds (or removes) relative to PTX is important for performance work:

| Concept | In PTX | In SASS |
| --- | --- | --- |
| Register names | Virtual (unlimited) | Physical (limited; spill if exceeded) |
| Instruction scheduling | Implicit / unordered | Explicit stall counts and barriers |
| Register reuse | Not expressed | Encoded in instruction control bits |
| Predication | `.pred` registers | Hardware predicate registers (PR) |
| Memory spaces | Named (`.global`, `.shared`) | Encoded in instruction opcode |
| Warp synchronization | `bar.sync` | SASS `BAR` family |
| New HW features (wgmma, TMA) | Via `sm_90a` PTX extensions | Native SASS opcodes |

## Why It Matters for AI Infrastructure

- **Operator performance**: When a custom CUDA kernel (e.g., a fused attention kernel or custom quantization op) is not reaching peak throughput, the first diagnostic step is to inspect its SASS for issues like register spilling, suboptimal memory access, or poor instruction-level parallelism.
- **Compiler trust**: PTX is the contract between `nvcc`/Triton/XLA and the hardware. Understanding this boundary clarifies what optimizations compilers can and cannot guarantee.
- **Hardware feature access**: New hardware features (e.g., Hopper's wgmma instructions, TMA — Tensor Memory Accelerator) often appear in SASS before they are cleanly exposed at the PTX or CUDA intrinsic level, requiring inline PTX or careful compiler targeting. The gap between hardware availability and clean software abstraction is often 6–18 months.
- **Reproducibility and debugging**: JIT compilation from PTX means the same `.ptx` binary may produce different SASS on different driver versions. This can cause subtle performance regressions or correctness issues in production. Pinning driver versions is a common mitigation in production ML infrastructure.
- **Register pressure and occupancy**: SASS inspection is the definitive way to determine actual register usage per thread. High register counts reduce occupancy (fewer concurrent warps per SM), which can hide latency and reduce throughput. Tools like `ptxas -v` report register usage, but SASS disassembly reveals whether the compiler chose wisely.
- **Debugging compiler bugs**: Both `ptxas` and Triton's LLVM backend can produce incorrect SASS in edge cases. Comparing expected behavior at the PTX level against observed SASS is often the only way to isolate such bugs.
- **Quantization and custom numerics**: Low-level quantization kernels (e.g., FP8, INT4, NF4) often require careful instruction selection at the SASS level to ensure correct rounding, correct data type conversions, and avoidance of unnecessary precision promotion. Verifying this requires SASS inspection.

## Common SASS Debugging Patterns

### Identifying Register Spills

Register spills appear in SASS as `STL` (store to local memory) and `LDL` (load from local memory) instructions. Local memory is cached in L2 but is significantly slower than register file access. Seeing these in a hot loop is a strong signal to reduce register usage — either by restructuring the kernel, splitting into multiple passes, or reducing thread-level parallelism.

```text
// Spill in SASS looks like:
STL [R1+0x4], R23 ;    // spill R23 to local memory at offset 4
...
LDL R23, [R1+0x4] ;    // reload R23 from local memory
```

### Reading Instruction Control Codes

In `nvdisasm` output (Ampere and earlier), each instruction is preceded by a control code line that encodes scheduling information:

```text
/*0010*/                   {
                            /* 0x0000000000000f00 */
                            /* stall=0, yield=0, read_bar=7, write_bar=7, wait_bar=0b00000000 */
                          }
/*0010*/        LFMA R4, R2, R4, R6 ;
```

High stall counts in a hot loop indicate a bottleneck — often a long-latency memory operation or a data dependency the scheduler cannot hide.

### Verifying Memory Access Patterns

SASS load/store instructions encode cache policy, address space, and access width. Verifying that global loads use `.U` (uniform) or `.E` (evict-first) modifiers as intended requires SASS inspection, since PTX-level cache hints may be rewritten by `ptxas`.

## Key Tools

| Tool | Purpose |

| `nvcc --ptx` | Dump PTX from CUDA source |

| CuAssembler | Community SASS assembler for sm_75–sm_86 |
| `kernel.asm['ptx']` | Triton Python API to inspect generated PTX |
| `kernel.asm['sass']` | Triton Python API to inspect generated SASS |

## Hopper-Specific Notes (sm_90a)

Hopper introduced new instruction classes that are notable enough to call out separately:

- **wgmma (Warpgroup Matrix Multiply Accumulate)**: A new asynchronous matrix multiply instruction that operates at the warpgroup (4-warp) level rather than the warp level. It has no PTX equivalent in the standard `wmma` family; it requires `sm_90a` PTX targets and is exposed via CUTLASS's CuTe abstractions or inline PTX.
- **TMA (Tensor Memory Accelerator)**: A hardware unit that handles bulk asynchronous data movement between global and shared memory, including multi-dimensional strided access and automatic tiling. TMA operations appear in SASS as `CP.ASYNC.BULK.TENSOR` variants.
